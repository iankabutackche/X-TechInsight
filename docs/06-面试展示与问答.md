## 一、3 分钟介绍稿（可直接练）

### 【0:00～0:30】项目是什么

> 面试官您好，我介绍一个项目叫 **X-TechInsight**，是一个面向 **技术文档** 的 RAG 问答平台。  
> 用户可以上传 PDF、Markdown 或 TXT，系统把文档切分、向量化存进向量库；提问时先 **检索相关片段**，再交给大模型 **基于片段** 生成回答，并且标注 **[Source N] 来源**，资料不足时会 **明确拒答**，避免幻觉。

### 【0:30～1:30】架构与亮点

> 架构上是 **前后端分离**：前端 React 三栏 UI，后端 FastAPI，对话和消息存在 **SQLite**，文档片段向量存在 **ChromaDB**。  
> 我重点做了三件事：  
> 第一，**领域标签过滤**——每个对话有一个 tag，比如「后端」或「金融」，检索时只查同 tag 的片段，减少跨领域串答。  
> 第二，**流式体验**——问答接口用 **SSE** 推 token，前端逐字显示。  
> 第三，**工程化**——上传用 BackgroundTasks 后台入库不阻塞；部署在 Render，处理了 CORS、Vite 构建环境变量、免费 tier 休眠等问题。

### 【1:30～2:30】一个具体例子

> 比如我上传一份项目 README，tag 选「后端」，问「项目用了哪些技术栈」。  
> 系统会从向量库检索 Top-4 相关片段，拼进 Prompt，GPT 回答时会引用 Source 1、Source 2。  
> 如果我没上传文档，或者问题和文档无关，就会返回「根据现有资料，我无法回答」——这是 Prompt 里 **硬性规定** 的。

### 【2:30～3:00】总结与展望

> 这个项目让我完整走通了 **RAG 全流程**，而不是只会调 ChatGPT API。  
> 当前是 MVP，没有做多用户登录；如果上线我会加 JWT、按 user_id 隔离对话和向量库。  
> 项目已 **公网部署**，也可以看 GitHub 代码。以上是我的介绍，谢谢。

## 二、高频面试题与参考答案

### Q1：什么是 RAG？你的项目里 RAG 链路是怎样的？

**答：**

RAG = Retrieval Augmented Generation，检索增强生成。

链路四步：

1. **Index**：上传文档 → loader 切分 → OpenAI Embedding → 写入 ChromaDB  
2. **Retrieve**：用户提问 → 向量相似度检索 Top-K（默认 4）→ 可选 domain 过滤  
3. **Augment**：把片段格式化成 `[Source N]` 块，拼进 Prompt  
4. **Generate**：GPT 基于 System Prompt + 片段 + 问题生成回答，SSE 流式返回  

---

### Q2：为什么用 RAG 而不是微调？

**答：**

技术文档场景下，文档会 **增删改**，RAG 上传新文件就能更新知识，成本低、可解释。  
微调需要训练数据和算力，且难以引用到具体段落。  
我的目标是 **可溯源的文档问答**，RAG 更合适。微调更适合固定风格或固定领域且数据稳定的任务。

---

### Q3：Top-K 怎么选？太大太小有什么问题？

**答：**

K 太小可能漏掉关键信息；K 太大 Prompt 变长、噪音多、延迟和成本上升。  
我默认 **K=4**，并在 `eval_test.py` 里做过不同 K 值的对比实验。  
生产环境还可以加 **Rerank** 二次排序，这是后续优化方向。

---

### Q4：怎么减少幻觉？

**答：**

三层控制：

1. **检索层**：只提供 Top-K 片段，不喂全库  
2. **Prompt 层**：System Prompt 规定只能用片段、必须标注 Source、无法回答时必须拒答  
3. **产品层**：前端展示 sources，用户可核对原文  

没有检索结果时，代码直接返回拒答，不调用模型「硬编」。

---

### Q5：domain 标签过滤是怎么实现的？

**答：**

上传文档时把对话的 tag 写入每个 chunk 的 metadata `domain` 字段。  
检索时 ChromaDB 使用 `filter={"domain": chat.tag}`，只在同领域片段里做 similarity search。  
前端还有 **挂载勾选** 和 **领域冲突提醒**，避免用户混挂不相关文档；后端检索目前主要按 tag 过滤。

---

### Q6：SSE 和 WebSocket 有什么区别？为什么你用 SSE？

**答：**

SSE 是 HTTP 单向流，服务器推、客户端读，实现简单，和 `fetch` + `ReadableStream` 配合即可。  
WebSocket 双向，适合实时协作、游戏。  
AI 逐 token 输出是 **单向推送**，SSE 足够，FastAPI 的 `StreamingResponse` 原生支持。

---

### Q7：上传大文件会不会阻塞 API？

**答：**

不会阻塞主请求。`/files/upload` 先把文件存盘，立即返回；  
解析、切分、向量化放在 **BackgroundTasks** 里异步执行。  
前端提示「正在后台解析」，用户可继续操作。

---

### Q8：SQLite 和 ChromaDB 为什么分开？

**答：**

职责不同。SQLite 存 **结构化** 数据：对话 id、名称、tag、消息历史，适合 CRUD。  
ChromaDB 存 **高维向量**，专做相似度检索，不适合当聊天数据库。  
这是典型的 **OLTP + 向量检索** 分工。

---

### Q9：为什么没做登录？多用户怎么做？

**答：**

MVP 阶段刻意聚焦 RAG 核心链路，登录属于用户体系，和 RAG 解耦。  
二期方案：`users` 表 + JWT 鉴权；`chats.user_id` 外键；Chroma 按用户或租户分 collection。  
不做登录 **不是遗漏**，是优先级取舍，面试时我会主动说明边界和扩展方案。

---

### Q10：部署遇到了什么问题？怎么解决的？

**答：（结合真实经历，选 2～3 个讲）**

1. **CORS**：前端 onrender.com 调后端，浏览器拦截；在后端配 `CORS_ORIGINS` 并加 onrender 正则兜底  
2. **VITE_API_BASE_URL**：构建时未注入，请求 URL 变成字面量；Render 配环境变量 + Clear cache redeploy；代码里加 fallback  
3. **Render 休眠**：免费 tier 冷启动 30～60 秒；演示前提前访问 `/health` 唤醒  
4. **OpenAI 429**：本地开发余额不足；充值并理解 rate limit  

细节见 `04-常见问题与排错.md`。

---

### Q11：ChromaDB 和 Pinecone / Milvus 怎么选？

**答：**

ChromaDB 适合 **本地 MVP**：零运维、和 LangChain 集成快、演示够用。  
Pinecone / Milvus 适合 **生产大规模**：分布式、高可用、多租户。  
如果上生产，我会考虑 pgvector（PostgreSQL 扩展）统一关系数据和向量，减少组件数。

---

### Q12：如果检索结果相关但模型仍答错怎么办？

**答：**

可能原因：片段被切断丢失上下文、K 不够、Prompt 约束不足。  
优化方向：

- 调 chunk_size / overlap  
- 提高 K 或加 Rerank  
- 加强 Prompt（要求逐步引用）  
- 加 **人工反馈** 标记错误回答，迭代检索策略  

---

### Q13：Render 免费版有什么限制？生产怎么改？

**答：**

主要限制：**休眠**、**磁盘不持久**（SQLite/Chroma 可能丢）、资源有限。  
生产会换：付费实例 + 持久化盘或 RDS；向量库上云；加鉴权、限流、监控。  
免费 tier 适合 **portfolio 演示**，不适合真实多用户生产。
