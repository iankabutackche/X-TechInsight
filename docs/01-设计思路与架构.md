## 一、项目定位

**X-TechInsight** 是一个面向技术文档的 **RAG（检索增强生成）问答平台**。

用户上传 PDF / Markdown / 纯文本，系统把文档切分、向量化存入本地向量库；提问时先检索相关片段，再交给大模型生成回答，并标注 `[Source N]` 来源。

### 核心目标

| 目标 | 说明 |
|------|------|
| 减少幻觉 | 只允许根据检索到的片段回答，资料不足时明确拒答 |
| 多对话并存 | 不同主题/领域用独立对话空间，互不干扰 |
| 领域隔离 | 用「领域标签」缩小向量检索范围，降低跨领域串答 |
| 可演示、可部署 | 本地开发 + Render 公网部署，适合 portfolio 展示 |

### 刻意不做的事（MVP 边界）

| 不做 | 原因 |
|------|------|
| 用户注册 / 登录 | MVP 聚焦 RAG 链路；多用户属于二期（JWT + `user_id` 隔离） |
| 模型微调 | 技术文档场景用 RAG 即可更新知识，成本低、可解释 |
| 扫描版 PDF OCR | 复杂度高，当前只支持可选中文字的 PDF |

## 二、技术选型与理由

| 层级 | 选型 | 为什么 |
|------|------|--------|
| 前端 | React + Vite | 组件化三栏 UI，Vite 开发快、构建简单 |
| 后端 | FastAPI | 原生 async、自动 OpenAPI 文档、SSE 流式友好 |
| 关系库 | SQLite | 对话/消息量小，零配置，适合 MVP |
| 向量库 | ChromaDB（本地持久化） | 轻量、与 LangChain 集成成熟 |
| AI 框架 | LangChain + OpenAI | Embedding + Chat 一条龙，社区资料多 |
| Embedding | OpenAI Embeddings | 与 GPT 同一生态，效果稳定 |
| 生成模型 | gpt-4o-mini | 成本低、响应快，演示够用 |
| 部署 | Render（前后端） | 免费 HTTPS，适合学生/个人项目 |

## 四、模块划分

### 4.1 前端（`frontend/`）

| 文件 | 职责 |
|------|------|
| `src/App.jsx` | 三栏主界面：对话列表、聊天、右侧配置（标签/上传/挂载） |
| `src/api.js` | 封装所有后端 API；生产环境读 `VITE_API_BASE_URL` |
| `src/MessageMarkdown.jsx` | 助手回复 Markdown 渲染 + 代码高亮 |
| `vite.config.js` | 本地开发代理 `/api` → `127.0.0.1:8000` |

**UI 设计要点：**

- **领域标签（tag）**：创建对话时自选（如「后端」「金融」），右侧可随时修改
- **知识库挂载**：全局资源池 + 按对话勾选；未挂载的文档不参与该对话的冲突检测
- **领域冲突提醒**：当前对话 tag 与已挂载文档 tag 不一致时，红色提示用户

### 4.2 后端 API 层（`backend/main.py`）

| 接口 | 方法 | 作用 |
|------|------|------|
| `/health` | GET | 健康检查，部署探活 |
| `/chats/create` | POST | 创建对话（name + tag） |
| `/chats/list` | GET | 获取对话列表 |
| `/chats/{id}/messages` | GET | 获取某对话历史消息 |
| `/chats/{id}` | PATCH | 更新对话名称或 tag |
| `/files/upload` | POST | 上传文档，**后台异步**切分入库 |
| `/query/stream` | POST | SSE 流式问答 |

**工程细节：**

- **CORS**：允许本地 Vite + Render 前端域名跨域
- **BackgroundTasks**：上传后立即返回，解析/向量化在后台执行，不阻塞 HTTP
- **SSE**：`event: token / sources / done / error`，前端逐字显示回答

### 4.3 数据层（`backend/database.py`）

```
chats                          messages
├── id (UUID)                  ├── id
├── name                       ├── chat_id  → chats.id
├── tag（领域标签）             ├── role (user / assistant)
└── created_at                 ├── content
                               └── created_at
```

- 每个 **ChatID** 独立存储消息，实现「对话空间隔离」
- 删除对话时级联删除其消息（`CASCADE`）

### 4.4 RAG 引擎

#### `loader.py` — 文档加载与切分

- 支持：`.pdf`、`.md`、`.txt`
- PDF 用 PyMuPDF 按页提取文本；MD/TXT 直接读取
- RecursiveCharacterTextSplitter 切分，默认 chunk 约 800 字、overlap 100
- 每个片段注入 metadata：`source_file`、`page_index`、`domain`（来自上传时的 tag）

#### `vector_manager.py` — 向量库管理

- ChromaDB 持久化目录：`backend/data/chroma/`
- 上传时：`add_documents()` 写入向量
- 问答时：`similarity_search(query, k=Top-K, filter={"domain": tag})`
- **领域过滤**：只检索与当前对话 tag 一致的片段，避免金融文档回答后端问题

#### `agent_logic.py` — 问答逻辑

- 默认 **Top-K = 4**
- System Prompt 强制规则：
  1. 只能用参考片段回答
  2. 资料不足 → 「根据现有资料，我无法回答这个问题。」
  3. 关键结论标注 `[Source N]`
- 返回：`answer` + `sources[]`（文件名、页码、摘要）

## 五、核心数据流（三条主路径）

### 5.1 上传文档

```
用户选文件 + tag
    → POST /files/upload
    → 保存到 backend/data/uploads/
    → BackgroundTasks 启动
        → loader 切分
        → vector_manager 向量化写入 ChromaDB
    → 立即返回 { status: "processing" }
```

### 5.2 提问（RAG 问答）

```
用户输入问题（当前 chat_id 已有 tag）
    → POST /query/stream
    → agent_logic.query(question, domain=chat.tag)
        → vector_manager 按 domain 检索 Top-K 片段
        → 拼装 Prompt（System + 参考片段 + 用户问题）
        → ChatOpenAI 流式生成
    → SSE 推送 token → 前端逐字显示
    → SSE 推送 sources → 前端展示溯源
    → 写入 SQLite messages（user + assistant）
```

### 5.3 加载对话历史

```
打开页面 / 切换对话
    → GET /chats/list
    → GET /chats/{id}/messages
    → 前端渲染历史（助手消息走 Markdown 组件）
```

## 六、关键设计决策（面试常问）

### Q1：为什么用 RAG，不用微调？

| RAG | 微调 |
|-----|------|
| 上传新文档即可更新知识 | 需要训练数据 + GPU + 周期 |
| 可溯源到具体片段 | 黑盒，难以引用来源 |
| 适合「资料会变的文档问答」 | 适合「固定风格/固定领域」 |

本项目是**技术文档分析**，文档会增删改，RAG 更合适。

### Q2：为什么用 domain 标签过滤，而不是全库检索？

全库检索容易把**无关领域**的相似表述检索进来（例如两个项目都有「接口设计」字样）。  
用 `filter={"domain": tag}` 在向量检索阶段缩小范围，**提高准确率、降低串域**。

### Q3：为什么 SQLite + ChromaDB 分开存？

| 存储 | 存什么 | 原因 |
|------|--------|------|
| SQLite | 对话结构、消息文本 | 关系清晰、查询简单 |
| ChromaDB | 文档片段向量 | 专做相似度检索，不适合当聊天历史库 |

职责分离，各用所长。

### Q4：为什么 SSE 流式，而不是等整段返回？

- 用户体验：首字快，像 ChatGPT
- 长回答不必干等
- FastAPI `StreamingResponse` 原生支持

### Q5：为什么 MVP 不做登录？

登录是**用户体系**，RAG 是**知识问答体系**，解耦开发。  
二期方案：`users` 表 + JWT；`chats.user_id`；Chroma collection 按租户隔离。

## 七、已知局限（诚实写进面试）

1. **Render 免费档**：15 分钟无访问休眠；磁盘不永久，重启可能丢 SQLite/Chroma 数据
2. **挂载 UI vs 检索**：前端按对话勾选挂载，后端检索目前主要按 **tag/domain** 过滤，尚未精确到「仅检索勾选文件 ID」
3. **单租户**：无登录，任何人拿到网址都能用（演示 OK，生产需鉴权）
4. **PDF 限制**：扫描版、加密 PDF 不支持或会报错
5. **OpenAI 依赖**：需 API Key 与网络；国内本地开发常需代理
