> 适合：开发时查阅、面试前复习、写文档时对照代码。

---

## 一、先建立整体图景

```
用户提问
   ↓
① 检索（Retrieval）→ 从向量库找最相关的文档片段
   ↓
② 增强（Augmentation）→ 把片段塞进 Prompt
   ↓
③ 生成（Generation）→ 大模型根据片段写回答
   ↓
流式返回 + 标注 [Source N]
```

这就是 **RAG** 三个字母的含义：**R**etrieval **A**ugmented **G**eneration（检索增强生成）。

---

## 二、RAG 是什么？和 ChatGPT 直接问有什么区别？

| 方式 | 知识从哪来 | 优点 | 缺点 |
|------|------------|------|------|
| 直接问 ChatGPT | 模型训练时「记住」的通用知识 | 开箱即用 | 不知道你的私有文档；可能瞎编 |
| RAG（本项目） | 你上传的文档片段 | 可更新、可溯源、可拒答 | 需要上传、切分、建索引 |

**类比：** ChatGPT 像「什么都略懂一点的顾问」；RAG 像「只许翻你给的文件夹再回答的顾问」。

## 三、文档怎么处理？（loader.py）

### 3.1 加载

| 格式 | 工具 |
|------|------|
| `.pdf` | PyPDFLoader（按页提取文本） |
| `.md` / `.txt` | TextLoader |

**限制：** 扫描版 PDF（图片）提取不到文字；加密 PDF 会报错。

### 3.2 切分（Chunking）

长文档不能整篇塞进 Prompt（有长度限制，且检索不精确）。  
所以用 **RecursiveCharacterTextSplitter** 切成小块：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `chunk_size` | 1000 字符 | 每块大约多长 |
| `chunk_overlap` | 150 字符 | 相邻块重叠，避免句子被拦腰切断 |

**Markdown 特殊处理：** 代码块前后加占位符，降低「代码块被切两半」的概率。

### 3.3 元数据（Metadata）

每个片段会带上：

| 字段 | 示例 | 用途 |
|------|------|------|
| `source_file` | `README.md` | 溯源显示文件名 |
| `page_index` | `3` | PDF 页码 |
| `domain` | `后端` | 向量检索过滤 |
| `domain_tags` | `后端` | 标签字符串形式 |

## 四、向量与 Embedding 是什么？

### 4.1 为什么普通搜索不够？

用户问：「项目用了哪些技术栈？」  
文档里可能写的是：「前端 React，后端 FastAPI……」

关键词搜索对 **同义表述** 不友好；**向量检索** 比较的是「语义相似度」。

### 4.2 Embedding 做什么？

把一段文字变成 **一串数字（向量）**，语义相近的文字，向量距离更近。

```
"技术栈有哪些"  →  [0.12, -0.34, 0.56, ...]
"用了什么框架"  →  [0.11, -0.33, 0.55, ...]   ← 距离近
"今天天气怎样"  →  [0.89, 0.02, -0.77, ...]  ← 距离远
```

本项目用 **OpenAI Embeddings** API 完成向量化。

### 4.3 Top-K 检索

问一个问题，从向量库找出 **最相似的 K 条** 片段（默认 **K = 4**）。

K 太小 → 可能漏掉关键信息  
K 太大 → Prompt 变长、噪音变多、成本变高

项目里有 `eval_test.py` 可对比不同 Top-K 效果（本地实验用）。

## 五、ChromaDB 向量库（vector_manager.py）

### 5.1 它存什么？

- **不是** 存整篇 PDF 文件  
- **是** 存：片段文本 + 向量 + metadata

持久化目录：`backend/data/chroma/`

### 5.2 领域过滤

```python
search_kwargs["filter"] = {"domain": domain}
```

只检索 `domain` 字段等于当前对话 tag 的片段。  
例如对话 tag =「后端」，就不会检索到 tag =「金融」的片段。

### 5.3 和 SQLite 的分工

| 库 | 存什么 |
|----|--------|
| SQLite | 对话列表、聊天消息（结构化） |
| ChromaDB | 文档片段向量（相似度检索） |

## 六、大模型与 Prompt（agent_logic.py）

### 6.1 模型

- 默认：**gpt-4o-mini**
- `temperature=0` → 输出更稳定、少随机发挥

### 6.2 System Prompt 四条铁律

1. 只能用参考片段回答  
2. 资料不足 → 必须说「根据现有资料，我无法回答这个问题。」  
3. 关键结论标注 `[Source N]`  
4. 回答简洁、结构化  

### 6.3 为什么不微调？

| 微调 | RAG |
|------|-----|
| 改模型权重，训练成本高 | 改文档即可更新知识 |
| 难以引用具体段落 | 天然带 Source 溯源 |
| 适合固定风格/任务 | 适合文档频繁更新的场景 |

## 七、FastAPI 后端要点

### 7.1 REST API

浏览器/前端用 HTTP 调后端：

| 方法 | 路径 | 作用 |
|------|------|------|
| POST | `/chats/create` | 建对话 |
| GET | `/chats/list` | 列表 |
| POST | `/files/upload` | 上传 |
| POST | `/query/stream` | 流式问答 |

本地可打开 http://127.0.0.1:8000/docs 看自动生成的 API 文档。

### 7.2 BackgroundTasks（异步上传）

上传文件时：

1. 立刻保存文件到磁盘  
2. HTTP 马上返回「已接收，后台解析中」  
3. **后台**再切分、向量化  

用户不用干等 30 秒，体验更好。

### 7.3 CORS

前端在 `5173` 或 `onrender.com`，后端在 `8000` 或另一个域名 → **跨域**。  
后端 `CORSMiddleware` 允许指定前端域名访问。

### 7.4 SQLite + SQLAlchemy

两张表：

- `chats`：id, name, tag, created_at  
- `messages`：chat_id, role, content, created_at  

`chat_id` 外键 + `CASCADE` 删除 → 删对话时消息一起删。

## 八、SSE 流式输出

### 8.1 是什么？

**SSE = Server-Sent Events**，服务器 **单向** 持续推数据给浏览器。

和 WebSocket 区别：SSE 更简单，适合「AI 逐字输出」这种场景。

### 8.2 本项目的事件类型

| event | 含义 | 数据示例 |
|-------|------|----------|
| `token` | 又生成了一小段文字 | `{"content": "项目"}` |
| `sources` | 引用来源列表 | `[{index, source_file, ...}]` |
| `done` | 结束 | `{}` |
| `error` | 出错 | `{"message": "..."}` |

### 8.3 前端怎么处理？

`api.js` 里 `fetch` + `response.body.getReader()` 读流，  
按 `\n\n` 切分 SSE 块，解析 `event:` 和 `data:` 行。

## 九、React 前端要点

### 9.1 三栏状态管理

用 React `useState` 管理：

- `chats` 对话列表  
- `activeChatId` 当前对话  
- `messagesByChatId` 各对话消息  
- `documents` 全局资源池  
- `mountedDocsByChatId` 各对话挂载了哪些文档  

### 9.2 Vite 代理（本地）

开发时 `api.js` 的 `API_BASE` 为空，请求 `/chats/list` 等路径，  
Vite 自动转发到 `http://127.0.0.1:8000`（见 `vite.config.js`）。

### 9.3 生产环境变量

`VITE_API_BASE_URL` 在 **build 时** 写入 JS，不是运行时读取。  
改 Render 环境变量后必须 **重新 Deploy**。
